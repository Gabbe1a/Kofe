**Версия:** 1.2  
**Дата:** 2026-07-09  
**Статус:** функционал по скринам IMG_1921…IMG_1974 (**54 PNG**) + зафиксированная инфраструктура (Яндекс.Облако, RuStore Push).  
**Цель:** воспроизвести **весь функционал** текущего приложения один в один; визуальный дизайн — по AURA (`teya-memory/design/`); поведение, поля, валидации, переходы и бизнес-правила — без потерь.

**Модель заказа (зафиксировано):** только самовывоз **«С собой»**. Доставки и зала в приложении нет.

## 0. Общие требования

### 0.1. Платформы
- iOS и Android (Flutter).
- Адаптация под safe area, home indicator, status bar.
- Поддержка тёмной темы как базовой (как в текущем приложении).

### 0.2. Навигация верхнего уровня
Постоянный **bottom navigation** из 4 вкладок (после онбординга):

| # | Вкладка | Иконка (смысл) | Экран |
|---|---------|----------------|-------|
| 1 | Профиль | силуэт человека | §6 |
| 2 | Меню | клош / блюдо | §5 |
| 3 | Корзина | корзина | §7 |
| 4 | Заведения | витрина / магазин | §8 |

Активная вкладка — акцентным цветом (в текущем приложении teal/cyan).  
Табы доступны всегда после выбора города и точки (или после восстановления сессии с сохранёнными city/venue).

### 0.3. Глобальное состояние сессии
Приложение обязано хранить и использовать:
- выбранный **город**;
- выбранную **точку** (venue);
- **тип заказа** (на скринах: «С собой»);
- профиль пользователя (имя, телефон, email, ДР, бонусы);
- корзину;
- сохранённые карты оплаты;
- опционально: сохранённый комментарий к заказу;
- флаг прохождения / закрытия promo-карусели (если показывается при старте).

### 0.4. Внешние действия (обязательные интеграции)
- `tel:` — звонок в точку / поддержку;
- `mailto:` / системный compose — поддержка по email;
- `https://` — франшиза (Google Form), амбассадоры, VK, Instagram, сайт coffee-mama.ru, legal docs;
- карты (Apple Maps / Google Maps / аналог / Yandex MapKit — выбрать на этапе карт) с кастомными пинами бренда;
- **push-уведомления:** Android — **RuStore Push** (основной канал в РФ); опционально HMS/FCM через Universal SDK; iOS — APNs.  
  Обязателен in-app inbox «Уведомления» независимо от доставки пуша.  
  Статус консоли: приложение RuStore **«Кофе»** создано; ключи Push — на этапе Flutter-сборки (`docs/DECISIONS.md`).

### 0.4.1. Backend / данные (зафиксировано)
- Прод: **PostgreSQL в Яндекс.Облаке** + API-слой.
- Клиент хранит локально только кэш/сессию/черновик корзины.
- Staff (бариста): **web-панель**, не отдельное мобильное приложение на старте.
- Детали схемы и статусов: `docs/architecture-backend-data.md`.

### 0.5. Принцип покрытия
Каждый экран ниже описан как **функциональная спецификация 1:1** с текущим приложением.  
Скролл-состояния одного экрана (IMG_1924–1928, 1948–1950 и т.п.) **не являются отдельными экранами**, но подтверждают объём данных и UI.

### 1.2. Выбор города
**Скрины:** IMG_1922  
**ID:** `CitySelectScreen`

#### Назначение
Выбор города сети перед работой с меню и точками.

#### UI
- Заголовок по центру: **«Выберите город»**.
- Поле поиска:
  - иконка лупы слева;
  - placeholder: **«Поиск города»**.
- Список городов (минимум зафиксированные на скрине):
  1. Ростов-на-Дону
  2. Азов
  3. Сочи
- Разделители между строками.
- Без bottom nav.

#### Поведение
1. Ввод в поиск фильтрует список по подстроке (без учёта регистра).
2. Тап по городу:
   - сохраняет `cityId` / название;
   - переходит к §1.3 Выбор точки для этого города.
3. Пустой результат поиска — пустой список (без падения).
4. Повторный вход из профиля (§6.1 «Выбор города») открывает этот же экран; после выбора — возврат в профиль/меню с обновлённым городом и сбросом/перевыбором точки при необходимости.

#### Данные
```text
City { id, name }
```

#### Критерии приёмки
- [ ] Есть поиск и список.
- [ ] Все три города из скрина доступны (список расширяем с API).
- [ ] Выбор персистится между запусками.

### 1.3. Выбор точки самовывоза («где заберёте заказ»)
**Скрины:** IMG_1923, IMG_1924, IMG_1925, IMG_1926, IMG_1927, IMG_1928  
**ID:** `VenuePickerScreen`

#### Назначение
Выбор конкретной кофейни для самовывоза: карта + список.

#### UI — шапка
- Кнопка назад (chevron) — возврат к выбору города или предыдущему экрану.

#### UI — карта (верх)
- Интерактивная карта города (тёмная стилизация).
- Пины всех точек с логотипом «Кофе».
- Жесты: pan, zoom.
- Тап по пину = выбор соответствующей точки в списке (синхронизация).

#### UI — нижняя панель / список
- Чип/кнопка режима: **«С собой»** — единственный формат заказа (только самовывоз; доставки/зала в приложении нет).
- Вертикальный скролл списка точек. Каждая строка:
  - radio (выбрано / не выбрано);
  - короткое имя / улица (жирный);
  - полный адрес;
  - иконка телефона + номер (акцентный цвет, кликабельно);
  - иконка часов + график работы:
    - вариант A: `пн-пт: HH:MM-HH:MM` и отдельно `сб-вс: …`;
    - вариант B: единый `пн-вс: …`.
- Фиксированная CTA внизу: **«Выбрать»**.

- ул. Волкова, 5
- пр-т Буденновский, 88
- бульвар Комарова, 20
- пр-кт Космонавтов, 19А (рядом с остановкой)

- пр-кт Ворошиловский, 79
- Буденновский 50

#### Поведение
1. По умолчанию может быть предвыбрана первая / ближайшая / ранее выбранная точка.
2. Выбор radio или пина обновляет selected venue.
3. Тап по телефону → системный звонок (`tel:`).
4. «Выбрать»:
   - сохраняет venue в сессию;
   - выставляет тип заказа «С собой» (если это единственный режим на этом экране);
   - переходит в основное приложение (§5 Меню) и/или §2 Promo, если требуется.
5. Список длинный — обязателен скролл без обрезания данных (доказано IMG_1924–1928).
6. Кнопка «Выбрать» не должна перекрывать возможность прочитать последнюю видимую карточку (в редизайне улучшить hit-area, функционал сохранить).

#### Данные
```text
Venue {
  id, cityId,
  shortName, fullAddress,
  phone,
  hours: [{ daysLabel, open, close }],
  lat, lng
}
```

- [ ] Карта + список синхронизированы.

- [ ] Телефон кликабелен.

## 4. Общие компоненты заказа (шапка меню)

Используются на экране Меню (и влияют на Корзину).

### 4.1. Тип заказа
- Единственный формат сети в приложении: **самовывоз «С собой»**.
- **Доставки in-app нет.** **«В зале» / dine-in через приложение нет.**
- Клиент заказывает в приложении и забирает заказ в выбранной точке лично.
- В UI строка «С собой» может отображаться как текущий тип (с иконкой/chevron как на скринах), но **смены на другой тип заказа нет** — это не multi-mode checkout.
- Баннеры «доставка через Delivery / Яндекс Еда» — внешний маркетинг/партнёры, не заказ внутри приложения.

### 4.2. Адрес точки
- Под типом заказа: короткий адрес, напр. **«пр-кт Нагибина, 35а»**.
- Тап → повторный §1.3 Выбор точки (или аналог в контексте меню).

### 4.3. Поиск по меню
**Скрины:** IMG_1947 (поле на меню), IMG_1973 (результаты)

- Поле **«Поиск»** с иконкой лупы.
- Экран/режим поиска:
  - строка запроса + clear (X);
  - кнопка **«Отмена»** (возврат в меню);
  - live-фильтрация по мере ввода (пример: «Лат»).
- Результаты:
  - группировка по категориям/линейкам (заголовки CAPS);
  - строка: thumbnail + название (может обрезаться `…`) + цена справа;
  - один продукт может встречаться несколько раз с разными ценами (разные объёмы/SKU).
- Тап по результату → PDP выбранного SKU.
- Пустой запрос / нет совпадений — empty state.

### 4.4. Promo-баннеры в меню
Горизонтальные/вертикальные баннеры над категориями (карусель или список). Зафиксированы на скринах:
1. Доставка кофе в офис/дом от 20 мин (Delivery / Яндекс Еда) — IMG_1934.
2. Бонусная система: регистрация, начисление 5% от оплаченной рублями суммы после выдачи заказа; списание до остатка 1 ₽ — IMG_1935 и актуализированное бизнес-правило.
3. 100 баллов за сторис — IMG_1947.

Тап по баннеру → детальная promo-страница / внешняя ссылка / слайд карусели (как в текущем приложении).

### 4.5. Звонок в точку из меню
**Скрин:** IMG_1935  
Системный action sheet:
- **«Позвонить +7 (960) 466-88-87»** (номер может быть номером поддержки или точки — брать из API);
- **«Отменить»**.

2. **Ресторан** — адрес точки, chevron → §1.3.
3. **Время самовывоза** — выбранное время, chevron вниз → inline **wheel picker**:
   - колонка даты («Сегодня», «Пн 6 июля»…);
   - час;
   - минута.
   **Скрин:** IMG_1965.

#### UI — промокод и подтверждение адреса
4. **Промокод** — chevron → §7.3.
5. **«Подтверждаю выбранный мною адрес»**:
   - ❗️ / красный акцент;
   - toggle (default off).
   - Если off при попытке оплаты / при непустой корзине: красный баннер  
     **«Не заполнено обязательное поле: '❗️ Подтверждаю выбранный мною адрес'»**  
     и **«Оплатить» disabled** (IMG_1971–1972).

#### UI — комментарий
6. **«Оставить комментарий»** → §7.2 (на пустой корзине IMG_1957 видно; на заполненной может быть ниже fold — функционал сохранить).

#### UI — оплата
7. **Тип оплаты** — напр. «Карта 2056», chevron → §7.4.

#### UI — ошибки и CTA
- Пустая корзина: **«Вы ничего не выбрали»** + **«Оплатить 0.0₽»** disabled (IMG_1957).
- Guest: **«Вы не вошли в профиль»** + pay disabled (IMG_1965).
- Не подтверждён адрес: ошибка обязательного поля + pay disabled (IMG_1971).
- Ready: **«Оплатить {sum}₽»** active (нужны: login + items + addressConfirmed + payment method).

#### Поведение оплаты
1. Валидации по порядку: auth → не пусто → addressConfirmed → способ оплаты → точка доступна.
2. Создание заказа → эквайринг (Alfa-Bank / текущий провайдер).
3. Успех → заказ в истории со статусом **подтвержден** (далее цепочка статусов) + ориентир готовности **~15 мин** + очистка корзины.
4. Ошибка оплаты → сообщение, корзина сохраняется.
5. Статусы дальше меняет сотрудник точки; клиенту — пуши (§6.5).

#### Критерии приёмки
- [ ] Пустая / guest / non-empty / address validation состояния.
- [ ] Time wheel, promo, payment method, upsell, clear cart.
- [ ] Badge на табе.
- [ ] Pay disabled до выполнения обязательных условий.

## 10. Бонусная программа

Зафиксированные правила из UI/контента:
- Баланс отображается в профиле.
- Начисление: **5%** с заказа (баннер).
- Оплата заказа бонусами: **без процентного ограничения**, но к оплате рублями должен остаться минимум 1 ₽.
- Акция сторис: +100 баллов / сторис, до 3 раз в неделю.
- История начислений может приходить через уведомления/отдельный API (если появится экран — не противоречить).

В чекауте заложить выбор «списать бонусы» (если есть в текущем API; на IMG_1957 отдельной строки не видно — **проверить на живом приложении/API** и не потерять, если функция есть за пределами сделанных скринов).

## 12.1. Жизненный цикл заказа (зафиксировано)

```text
Pay success
  → status: confirmed
  → UI: ориентир готовности ~15 минут (таймер/время)
  → сотрудник точки вручную:
       confirmed → preparing → ready → issued
  → клиент получает push на каждый шаг (§6.5)
  → в истории финально: выполнен / выдан
```

Таймер 15 мин — ориентир для гостя; фактический статус всегда от точки.

## 14. Нефункциональные требования

1. **Оффлайн:** корректные ошибки сети, не молчаливый пустой экран.  
2. **Производительность:** карта и длинные списки без jank; изображения с кэшем.  
3. **Доступность:** достаточный контраст в редизайне; hit-area ≥ 44pt.  
4. **Локализация:** RU как основной язык.  
5. **Аналитика:** события screen_view + key actions (add_to_cart, purchase, select_venue…).  
6. **Безопасность:** токены в secure storage; маски карт; не логировать PAN.  
7. **Разрешения:** геолокация (заведения/locate me), уведомления, камера — только если реально нужно (для сторис-акции камера не обязательна внутри app).
